1. 리미트 스위치는 2개, 리미트스위치로 모터의 이동 범위 끝과 끝을 감지, 고장 감지로 사용하려고함

1번과 2번의 위치는 변경될 수 있음

NO가 1이면 NC가 0 인 SPDT의 한쪽이 1이면 한쪽이 떨어지는 리미트 스위치임

각 스위치의 NO와 NC의 값이 같으면 리미트 스위치 고장으로 판단 - 모터 정지 시퀀스

두 스위치의 값이 같으면 고장을 판단 - 정지

리미트 스위치가 눌려도 모터는 일정 범위만큼 더 가야함(설정된 위치(목표 위치)까지 가야함)

예를 들어 모터의 목표 위치 한쪽이 목표 1이라고하고 한쪽을 목표 2라고 할때, 
목표 1의 위치 근처에서 스위치 1의 NO가 1이 되지 않으면 고장 - 정지
목표 1의 위치를 충분히 벗어난 위치인데 스위치 1의 NO가 1이면 고장 - 정지
목표 2의 위치 근처인데 스위치 2의 NO가 1이 되지 않으면 고장 - 정지
목표 2의 위치를 충분히 벗어난 위치인데 스위치 2의 NO가 1이면 고장 - 정지

목표 위치와 리미트 스위치의 범위는 정해져있지 않음 - 데이터로 관리해야함

정지에 대한 시퀀스는 아직 미정(모터가 멈추고 브레이크를 채우는 시간, 브레이크를 채우고 모터를 멈추는 시간 등)


신호명은 아래와 같음(참고)

LIMIT_SWITCH1_NO
LIMIT_SWITCH1_NC
LIMIT_SWITCH2_NO
LIMIT_SWITCH2_NC



4. IPC 없이 공유 메모리 사용 시 필수 주의사항
volatile 키워드 필수 사용: 최적화 단계에서 컴파일러가 메모리 값을 레지스터에 가둬두고 재참조하는 것을 막기 위해, 공유 변수에는 반드시 volatile을 선언해야 바로 RAM에 반영됩니다.

데이터 오염 (Race Condition) 방지: 인터럽트(IPC 핸드셰이크)가 없기 때문에 CPU1이 데이터를 다 쓰기 전에 CM이 읽어갈 위험이 있습니다. 이를 방지하려면 데이터 배열의 마지막 바이트에 Status Flag(예: 0x55AA 등 완성 신호)를 두고, 읽는 쪽에서 플래그가 완료되었는지 체크한 뒤 가져가는 구조(폴링 방식) 링 버퍼 메커니즘을 소프트웨어적으로 구현해야 합니다.
